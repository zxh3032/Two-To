# frontend 开发协作规范

## 适用范围

本文件适用于 `frontend/` 目录下的所有前端代码、测试、静态资源和工程配置。后续使用大模型修改前端时，优先遵守本文档；如果本文档与更深层目录中的 `AGENTS.md` 冲突，以更深层目录文档为准。

## 项目概览

已确认：

- `frontend` 是 Two-To 的 Web 前端工程目录。
- 项目产品方向是宠物适配匹配、品种解析、宠物档案、健康照料、日常管理与成长记录。
- 根 README 约定前端技术方向为 Vite + React + TypeScript + React Router + TanStack Query，Web 端优先。
- 当前已存在 `package.json`、Vite 配置、React 入口、路由、全局布局、后端 ping 联通检测和 slogan 展示入口。

推断：

- 前端会围绕“相遇与相知”和“相守与照料”两段用户旅程建设页面与功能模块。
- 当前分层设计目标是 Feature-Sliced 风格的轻量变体：应用入口、页面、功能模块、领域实体、共享能力逐层依赖。

## 协作与需求确认规则

本项目是个人项目，很多产品想法、技术文档或代码调整会以一句话或一小段描述出现。大模型协作开发时，不能把这类描述直接当作完整规格执行。

要求：

- 执行任何前端代码修改前，必须先说明准备如何改、涉及哪些文件、预期效果和主要风险，等用户明确同意后再落地。
- 如果用户已经明确说“直接改”“开整”“按这个做”等，可视为对当前已说明范围的同意；一旦发现新增范围、交互歧义或实现风险，需要再次停下来确认。
- 遇到功能目标、用户场景、交互路径、文案语气、视觉风格、接口语义等不确定点时，先提出具体问题和可选方案，不要自行脑补后继续实现。
- 涉及需求相关内容时，需要主动和用户讨论目标用户、核心场景、边界条件、优先级和可能取舍；不要简单按字面照单执行。
- 用户不要求专业术语描述，沟通时优先使用清楚的中文解释方案、差异和影响。

## 视觉设计基调

前端页面由大模型承担 UI/UX 方案设计责任。用户不熟悉前端、UI 和交互规范时，不能把设计决策反推给用户；需要结合 `ui-ux-pro-max` 的设计检查方法主动给出可执行方案。

Two-To 的视觉方向：

- 整体以黄色、粉色等明亮、温暖的颜色为主，营造温馨、柔和、亲近的宠物陪伴感。
- 色彩可以搭配少量奶白、浅橙、浅桃粉、暖黄、柔和绿色等辅助色，避免页面变成单一色块或低对比度糖果色。
- 页面气质应符合宠物相关产品：友好、有生命力、轻松、可信赖，不做冰冷的企业后台风格，也不做过度娱乐化的花哨视觉。
- 组件圆角、阴影、插图、状态反馈可以更柔和，但仍需保证可读性、对比度、触控尺寸、键盘可访问和移动端无横向滚动。
- 具体页面或组件设计前，优先说明信息层级、核心动作、空态/加载/错误态、移动端布局和色彩使用，再请求用户确认。
- 如果后续引入图片、插画或图标，优先选择真实表达宠物、陪伴、照料、成长记录等主题的视觉资产；不要用无关装饰堆砌氛围。

## 整体架构

当前目录结构：

```text
frontend/
  public/               # 静态资源，构建时原样输出
  src/
    app/                # 应用初始化、路由、全局 Provider、全局布局
    pages/              # 路由页面，负责组装 feature 和 entity 展示
    features/           # 具体业务功能，例如适配测评、品种检索、宠物档案编辑
    entities/           # 领域实体，例如 user、pet、breed、assessment
    shared/
      api/              # 请求客户端、接口封装、错误处理
      config/           # 前端环境配置、常量配置
      lib/              # 通用工具函数、hooks、格式化逻辑
      proto/            # 由 backend/proto/ 生成的 TypeScript 请求/响应类型
      styles/           # 全局样式、主题变量、设计 token
      ui/               # 通用 UI 组件，例如 Button、Modal、FormItem
    assets/             # 图片、字体、图标等源码资产
  tests/                # 前端测试与测试工具
```

推荐依赖方向：

```text
app -> pages -> features -> entities -> shared
```

禁止反向依赖：

- `shared` 不依赖 `entities`、`features`、`pages`、`app`。
- `entities` 不依赖 `features`、`pages`、`app`。
- `features` 不直接依赖 `pages` 或 `app`。
- 不相关的 `features` 之间不直接互相调用；确实需要复用时，优先下沉到 `entities` 或 `shared`。

## 分层职责

### app

`src/app` 是应用装配层，负责 React 根节点、路由、全局 Provider、全局布局、错误边界和页面级 loading。

要求：

- 不写具体业务规则。
- 不直接拼接口请求参数。
- 不直接处理某个业务模块的数据转换。
- 后续如果引入状态管理、请求缓存、主题系统，统一从这一层接入。

### pages

`src/pages` 是路由页面层，负责组织页面布局并组合多个 `features` 或 `entities` 展示。

要求：

- 可以读取路由参数、设置页面标题、控制页面级 loading/empty/error 状态。
- 不沉淀复杂业务规则；复杂流程下放到 `features`。
- 不直接操作底层请求客户端；接口动作优先由对应 `feature` 或 `entity` 承接。

### features

`src/features` 是功能模块层，承载明确的用户动作或业务流程。

适合放置：

- 适配性测评提交与结果展示。
- 品种筛选、检索、收藏等动作。
- 宠物档案创建、编辑、删除确认等流程。
- 健康提醒创建、完成、列表筛选等流程。

要求：

- 可以依赖 `entities` 的领域类型、展示组件和基础逻辑。
- 可以依赖 `shared/api`、`shared/lib`、`shared/ui`。
- 不把无业务语义的通用组件长期留在 feature 内；确认多处复用后再下沉。

### entities

`src/entities` 是领域实体层，沉淀稳定业务对象及其基础展示能力。

适合放置：

- `pet`、`breed`、`user`、`assessment` 等领域对象。
- 多个功能都会使用的领域类型、基础展示组件、简单领域转换。
- 与领域强相关但不绑定某个页面动作的工具逻辑。

要求：

- 不依赖具体页面或功能流程。
- 不持有页面级状态。
- 不直接处理路由跳转。

### shared

`src/shared` 是共享基础层，只放与具体业务无关或弱相关的能力。

目录职责：

- `shared/api`：HTTP client、请求拦截、错误归一化、基础请求类型。
- `shared/config`：环境变量读取、路由常量、运行配置。
- `shared/lib`：通用 hooks、日期格式化、校验、轻量工具。
- `shared/proto`：从 `backend/proto/` 生成的 TypeScript 类型，不手写生成物。
- `shared/styles`：全局样式、主题变量、设计 token。
- `shared/ui`：无业务语义的基础 UI 组件。

要求：

- `shared/proto` 的源头是 `backend/proto/*.proto`，不要直接修改生成后的类型文件。
- `shared/ui` 不出现宠物、测评、品种等具体业务概念。
- `shared/lib` 不为一两行重复逻辑随意增加 helper。

## 核心调用链路

页面渲染链路：

```text
app 初始化
  -> pages 路由页面
    -> features 业务动作与流程
      -> entities 领域对象与展示能力
      -> shared/api 发起请求
        -> shared/proto 约束请求参数与返回值类型
```

新增接口联调链路：

```text
backend/proto/*.proto
  -> 后续生成 frontend/src/shared/proto/*.ts
    -> shared/api 定义请求方法
      -> features 调用请求方法并处理业务状态
        -> pages 组合页面展示
```

当前现实：

- `shared/proto/README.md` 已明确 TypeScript 生成工具尚未接入。
- 已接入 React Router 管理页面路由。
- 已接入 TanStack Query 管理异步请求状态。
- 已通过 `features/api-health-check` 打通首页到后端 `/api/v1/ping` 的联通检测链路。
- 已通过 `features/project-slogan` 打通首页到后端 `/api/v1/slogan` 的项目口号展示链路。

## 数据与模型设计

已确认：

- 前后端请求参数和返回值优先由 `backend/proto/` 统一定义。
- 前端生成类型目标目录是 `src/shared/proto`。

约定：

- 页面私有 UI 状态放在页面或 feature 内，不上升到领域层。
- 业务对象类型优先贴近使用场景；跨 feature 复用后再沉淀到 `entities`。
- API 入参和出参优先引用 `shared/proto` 的生成类型，避免前后端重复手写 DTO。
- 展示模型如果与后端响应不一致，可以在 feature 或 entity 内做明确转换，并用中文注释说明转换原因。

## 注释规范

新增或修改代码时必须补充必要中文注释。

要求：

- 方法、函数、复杂 hooks、复杂组件需要说明作用。
- 请求参数和返回值需要说明业务意义，尤其是 ID、状态、枚举、时间字段和分页字段。
- 对复杂流程写“大步骤”注释，不做逐行翻译式注释。
- 对兼容逻辑、兜底逻辑、异常分支要写清楚原因。
- 生成物不补手写注释；应修改源文件或生成脚本。

TypeScript 建议格式：

```ts
/**
 * 提交宠物适配测评答案，并返回本次测评的推荐结果。
 * @param payload 测评答案与用户基础偏好，字段来自 proto 生成类型。
 * @returns 后端计算后的推荐结果与解释文案。
 */
```

## 日志与排查规范

前端日志同样重要，但要避免泄露隐私数据。

关键位置需要有可排查信息：

- API 请求失败、超时、重试和降级。
- 登录态变化、权限拒绝、会话过期。
- 重要表单提交失败。
- 复杂 feature 的关键步骤切换。
- Error Boundary 捕获到页面级异常。

要求：

- 日志中包含模块名、动作名、关键业务 ID、错误码和错误摘要。
- 不打印完整手机号、token、地址、详细病历、备注等敏感内容。
- 浏览器端日志不能替代用户可见的错误状态；页面仍需展示明确的失败、空态或重试入口。
- 后续如果接入埋点或前端日志平台，需要把日志封装放在 `shared/lib` 或 `shared/api` 的基础层，但不要为了少量调用过早抽象。

## 封装与模块拆分原则

不要过度封装。

具体要求：

- 不因为一两行逻辑重复就抽 helper。
- 不把阅读时需要连续理解的业务流程拆成大量跳转函数。
- 只有在多处复用、命名清晰、能减少真实复杂度时，才下沉到 `shared/lib` 或 `entities`。
- UI 组件先就近放置；确认无业务语义且多处复用后，再移动到 `shared/ui`。
- 请求封装先贴近 feature；多个模块复用同一资源时，再考虑沉淀到 `entities` 或 `shared/api`。

## 增量修改与兼容要求

修改已有逻辑时，优先增量演进，不要一刀切删除。

要求：

- 旧逻辑如果需要替换，先注释保留，并写明“待验证后删除”或“兼容旧流程”。
- 新逻辑放在旧逻辑附近，便于 code review 对照差异。
- 对用户可见行为、接口参数、路由路径、数据结构的修改，要在注释或文档中说明兼容影响。
- 确认旧逻辑无调用、无兼容价值后，再单独提交清理改动。

## 开发建议

新增页面：

1. 在 `pages` 建页面入口。
2. 将页面动作拆到对应 `features`。
3. 将跨功能复用的业务对象放到 `entities`。
4. 将无业务语义的基础能力放到 `shared`。

新增接口：

1. 先在 `backend/proto/` 定义请求和响应。
2. 生成前端 TypeScript 类型到 `src/shared/proto`。
3. 在 feature 内实现请求调用和业务状态处理。
4. 多模块复用后再下沉 API 封装。

排查问题：

- 先看页面层是否正确传递路由参数。
- 再看 feature 是否正确处理 loading、empty、error、success 状态。
- 再看 `shared/api` 的请求参数、响应转换和错误归一化。
- 最后对齐后端 proto 与实际接口实现。

## 校验要求

当前前端工程已经提供 npm 脚本。修改前端代码至少执行：

- `npm run lint`
- `npm run test`
- `npm run build`
- 必要时补充浏览器截图或交互验证。

## 提交编号规则

前端提交编号独立计数，统一使用 `taro-xxxx` 格式，`xxxx` 为四位数字。

要求：

- 从本次提交开始，如果历史提交中没有前端编号，则第一条前端提交使用 `taro-0001`。
- 每次提交前，先查看历史提交中最近一次前端编号，再将编号加 1。
- 提交信息必须以编号开头，格式为 `taro-0001 中文提交描述`。
- 不复用已经出现过的编号，不跳号，数字不足四位时左侧补 `0`。
- 如果一次改动同时包含前端和后端，优先拆成两次提交，前端提交使用 `taro-xxxx`，后端提交使用 `tutu-xxxx`，两边分别计数。

查看前端最近编号的推荐命令：

```bash
git log --format=%s -- frontend | rg '^taro-[0-9]{4}' | head -1
```
